## 1. ОБЩАЯ ИДЕЯ ПРОЕКТА
**AI Meeting Translator** — это мобильное приложение для синхронного перевода, стенографирования и саммаризации рабочих встреч (митингов) в реальном времени.

*   **Главная задача**: Обеспечить пользователя текстовой транскрипцией и переводом речи участников встречи в реальном времени, а после завершения — предоставить краткое ИИ-резюме встречи (Summary), список задач (Action Items) и возможность интеллектуального семантического поиска по архиву встреч.
*   **Главный MVP-сценарий**: Запись аудио через микрофон мобильного устройства → нарезка на чанки → передача по WebSocket на FastAPI бэкенд → парсинг и перевод → вывод диалога в UI → загрузка архива встречи на сервер.
*   **Roadmap проекта**:
    1.  **MVP 1.0 (Текущая стадия)**: Реализована сквозная WebSocket трансляция PCM аудио от клиента к бэкенду с обратной отсылкой текстовых реплик по таймеру для проверки E2E шины. Разработан интерфейс Fluid Onyx.
    2.  **Интеграция ИИ-моделей**: Подключение к бэкенду конвейера WhisperX (для STT с диаризацией по спикерам) и LLM (Gemini/Llama через OpenRouter) для перевода и саммаризации.
    3.  **Локальный оффлайн-буфер**: Сохранение аудио-чанков в SQLite/File на случай разрыва связи с последующей довыгрузкой.

## 2. ОБЩАЯ АРХИТЕКТУРА И РОЛИ СЛОЕВ
Приложение состоит из мобильного клиента на Flutter и бэкенда на FastAPI. Взаимодействие происходит по гибридному протоколу: WebSocket для реалтайм-стриминга и REST API для архивных операций и управления сессиями.

### ASCII-схема архитектуры:
```
  [ Flutter Client ] <======================== WebSocket (Live Audio / Text JSON) ========================> [ FastAPI Backend ]
  (AudioRecorder PCM)                                                                                          (Uvicorn / FastAPI)
         |                                                                                                             |
         v (REST HTTP)                                                                                                 v
  [ Dio API Client ] ----------> POST /meetings/upload (Full Data Card) -----------------------------------> [ Routers & Core ]
                                                                                                                       |
                                                                                                         +-------------+-------------+
                                                                                                         |                           |
                                                                                                         v                           v
                                                                                                  [ Core Interfaces ]      [ State Machine ]
                                                                                                  (STT, Translate)        (CREATED->RECORDING)
                                                                                                         |                           |
                                                                                                         v (Mocks / Wrappers)        v
                                                                                                  [ MockSTT / OpenRouter ]    [ In-Memory DB ]
```

### Роли архитектурных слоев:
1.  **Flutter Client (UI/Logic)**: Захватывает звук с микрофона в PCM-формате, управляет локальным таймером встречи, отображает чат-ленту перевода, инициирует семантические поисковые запросы по архиву.
2.  **WebSocket Connection**: Дуплексный канал с низкой задержкой. Передает сырой аудиопоток на сервер и мгновенно возвращает распознанные реплики участников.
3.  **FastAPI Backend (API Gateway & Processing)**: Маршрутизирует запросы, реализует стейт-машину состояния встречи, обрабатывает входящий аудио-стрим.
4.  **STT & Translation Core**: Набор интерфейсов (`BaseSTTProvider`, `BaseTranslationProvider`), инкапсулирующих взаимодействие с Whisper и LLM.
5.  **Database & Storage (Supabase/R2)**: Сохраняет архивные встречи, текстовые стенограммы и полные аудиозаписи.

## 4. FRONTEND (FLUTTER)
Интерфейс спроектирован на основе жестких правил дизайн-системы Fluid Onyx (фоновый цвет `#0D0E12`, акценты `#7F3DFF`):

### Экраны (Screens):
1.  **HomeScreen**: Главный экран. Содержит круглую интерактивную кнопку начала митинга (диаметр 140) и симметричную 9-полосную звуковую волну с каждой стороны.
2.  **MeetingScreen**: Экран активной записи встречи. Отображает горизонтальный список участников со статус-волнами голоса, ленту сообщений от бэкенда и пунктирный таймлайн.
3.  **MeetingDetailScreen**: Экран просмотра результатов встречи. Содержит метаданные, плеер и встроенный поиск по стенограмме.
4.  **ArchiveScreen**: Список завершенных встреч с возможностью семантического AI-поиска и редактирования названий.

### Сетевые сервисы (Services):
*   **SocketService**:
    *   `connect(meetingId)`: Открывает WebSocket по адресу `ws://$host:8000/api/v1/ws/meeting/$meetingId`.
    *   `startRecordingAndStreaming()`: Инициализирует захват звука в PCM 16-bit, 16kHz, mono с помощью пакета `record` и отправляет чанки через `sink.add(Uint8List)`.
    *   Реализована обработка отключений: при потере сети сокет переподключается каждые 3 секунды.
*   **ApiService**:
    *   Выполняет HTTP-запросы через клиент `Dio` (загрузка карточки встречи по `/meetings/upload`, получение списка по `/meetings`).

### Поток данных при START и STOP:
*   **START**:
    Клик по Mic на HomeScreen → Переход на `MeetingScreen` → `SocketService.connect()` → `SocketService.startRecordingAndStreaming()` → микрофон захватывает PCM → сокет шлет пакеты → сокет принимает реплики → UI обновляет `List<SpeechSegment>` → `_scrollToBottom()`.
*   **STOP**:
    Клик на Stop на `MeetingScreen` → `SocketService.stopRecordingAndStreaming()` → закрытие стрима микрофона и WebSocket → `ApiService.uploadMeetingData()` сохраняет итоговую карточку в архиве.

## 5. BACKEND (FASTAPI)
Реализован на базе FastAPI и запускается через ASGI сервер `uvicorn`.

### Эндпоинты и API роуты:
*   **WebSocket**:
    *   `WS /api/v1/ws/meeting/{meeting_id}`: Стриминговый прием PCM-аудио и выдача текстовых реплик.
*   **REST API**:
    *   `POST /api/v1/meetings/start`: Инициализация новой записи встречи.
    *   `POST /api/v1/meetings/{meeting_id}/stop`: Остановка записи.
    *   `POST /api/v1/meetings/upload`: Загрузка полной карточки встречи с метаданными.
    *   `GET /api/v1/meetings`: Возвращает архив всех встреч.
    *   `GET /api/v1/meetings/search?q={query}`: Полнотекстовый поиск по архиву.

### Текущий статус бэкенд-сервисов:
*   `MockSTTProvider` (Заглушка): Имитирует генерацию оригинального текста.
*   `MockTranslationProvider` (Заглушка): Симулирует русскоязычный перевод.
*   `MockStorageProvider` (Заглушка): Имитирует загрузку файлов в S3/R2 хранилище.

## 9. МОДЕЛИ ДАННЫХ
*   **SpeechSegment (Фронтенд)**:
    *   Поля: `speakerInitials` (String), `timestamp` (String), `text` (String), `color` (Color).
    *   Жизненный цикл: Создается динамически в `MeetingScreen` при получении события из сокета, уничтожается при закрытии экрана.
*   **Meeting (Бэкенд / База данных)**:
    *   Поля: `id` (String), `status` (Enum: CREATED/RECORDING/NETWORK_LOST/FAILED/COMPLETED), `chunks` (Dict), `max_chunk_index` (Int).
    *   Жизненный цикл: Инициализируется при открытии WebSocket, меняет статусы во время записи, закрывается в статус `COMPLETED` по завершении.

---

## 10. ПОТОКИ ДАННЫХ (DATA FLOW)
*   **Audio Flow**: Микрофон устройства $\rightarrow$ PCM-пакеты $\rightarrow$ `SocketService` $\rightarrow$ WebSocket (бинарные данные) $\rightarrow$ `meeting_stream.py` (FastAPI).
*   **Text & Translation Flow**: Прием байтов бэкендом $\rightarrow$ логирование $\rightarrow$ сборка пакетов $\rightarrow$ эмуляция Whisper/Translate $\rightarrow$ отправка JSON с типом `segment` $\rightarrow$ WebSocket $\rightarrow$ Flutter `jsonDecode` $\rightarrow$ обновление `List<SpeechSegment>` $\rightarrow$ рендер списка.

## 12. ЧТО ЯВЛЯЕТСЯ ЗАГЛУШКАМИ (MOCKS)
1.  `MockSTTProvider`, `MockTranslationProvider` на бэкенде — возвращают тестовые фразы и имитируют STT-перевод.
2.  Локальный массив `MOCK_DB` на бэкенде — хранит сессии в оперативной памяти сервера без записи на диск.
3.  Имитация выгрузки файлов в `MockStorageProvider` (файлы не пишутся в S3 хранилище R2).
